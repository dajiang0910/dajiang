# 相似度检索

> 一句话：检索质量决定 RAG 上限——向量库搜不到对的片段，LLM 再强也白搭。相似度检索是 RAG 检索层的最后一环：把用户问题向量化 → 在向量库里找语义最近邻 → 返回 top-K 片段。

## 它解决什么问题

RAG 的核心链路：用户问 → **检索相关片段** → 拼入 Prompt → LLM 生成答案。检索这一环如果没把相关信息捞出来，后面的 LLM 就是"巧妇难为无米之炊"。

**不用检索评估会怎样？** 凭感觉调 top-K 和 threshold，上线后才发现"有时候答得很好，有时候牛头不对马嘴"——因为没有人告诉你哪些查询的检索是失败的。

## 核心要点

### 1. top-K 的权衡

| top-K | 优点 | 风险 | 适用场景 |
|-------|------|------|---------|
| 3-5 | 精准、成本低 | 可能漏掉分布在多个 chunk 的答案 | FAQ 式简单问答 |
| 8-10 | 覆盖面广 | 噪声多、context window 占用大 | 综述/总结类提问 |
| 20+ | 几乎不漏 | 噪声淹没信号、Prompt 超长易超 token 上限 | 一般不推荐 |

**面试关键补充**：top-K 和 precision 的关系——K 越大，后面的结果相似度越低，precision 单调递减。调 top-K 不是越大越好，而是要在"召回率"和"精确率"之间找平衡。

### 2. minScore（相似度阈值）

```
threshold=0.7 → 只要高置信度结果，精度高但可能漏召回
threshold=0.5 → 放宽门槛，多捞一些，但混入噪声
threshold=0.55 → 不存在"万能值"！要根据实际评估数据调
```

**Day 4 实验发现的铁律**：阈值调参**无法修复排序问题**。

```
"商品坏了怎么办" 的检索结果：
  Rank 1: refund-policy   score=0.5664  ← 错误答案分数更高
  Rank 2: shipping-policy score=0.5185  ← 正确答案分数更低

调 threshold=0.55 → refund 通过，shipping 被拦 → 更差！
调 threshold=0.5  → 两条都通过，但正确答案排 #2
```

**结论**：如果正确答案分数低于错误答案，调阈值只会先杀掉正确答案。排序问题需要混合检索 + rerank 来解决（Week 6）。

### 3. filterExpression 的坑（Spring AI 2.0.0 RedisVectorStore）

**期望**：`filterExpression: "source == 'shipping-policy'"` → 只在物流配送文档里检索

**实际**：不生效。0 命中。

**根因**：Spring AI 的 `RedisVectorStore` 把 metadata 序列化为 JSON 字符串存入单个 Redis Hash field：

```
Redis Hash 实际结构：
  content:   "[来源: shipping-policy] ..."
  metadata:  '{"source":"shipping-policy","chunk_index":0,...}'  ← JSON 字符串
  embedding: [0.12, -0.34, ...]

filterExpression 尝试匹配：
  source == 'shipping-policy' → 找独立的 "source" field → 不存在 ❌
```

RediSearch 的 FILTER 语法只能匹配独立 hash field，无法穿透 JSON。

**工程修复**：内容前缀后置过滤。既然 source 已经内嵌在 `[来源: shipping-policy]` 前缀中，检索后手动过滤：

```java
// 多取一些，用内容前缀匹配过滤
results.stream()
    .filter(doc -> doc.getText().contains("[来源: " + sourceFilter + "]"))
    .limit(topK)
    .toList();
```

面试加分点："Spring AI 的 metadata 序列化存在局限，我在灌库时把关键元数据内嵌到内容前缀，检索后自己做了后置过滤——不依赖框架的 filterExpression。"

### 4. precision@K：检索质量的量化标尺

```
precision@K = (top-K 结果中相关的条数) / K
```

Day 4 用 3 篇测试文档 + 5 个查询 + 人工标注，评估了检索质量：

| 查询 | precision | 发现 |
|------|-----------|------|
| Q1-Q3, Q5（简单查询） | 100% | query ≈ doc 原文词 |
| Q4 "商品坏了怎么办" | 0%（threshold 0.6） | query-doc gap 导致召回失败 |
| Q4（threshold 0.5） | 1/2 | 正确答案在，排第二 |

**关键发现**：query-doc gap 是检索失败的第一大杀手——用户用口语问"商品坏了"，原文写的是"运输途中损坏"，Embedding 没把这两个关联起来。

## Java 里怎么落地

### 项目代码（Week 5 Day 4）

**SearchService**：封装检索 + 格式化 + 后置过滤

```java
@Service
public class SearchService {
    // 基本检索
    public List<Document> search(String query, int topK, double threshold);

    // 带来源过滤（后置过滤，匹配内容前缀）
    public List<Document> searchWithFilter(String query, int topK, double threshold, String sourceFilter);

    // 检索 + 格式化为可拼 Prompt 的文本
    public SearchResult searchAndFormat(String query, int topK, double threshold);
}
```

**SearchController**：
- `POST /api/search` — 基本语义检索
- `POST /api/search/filtered` — 按来源过滤检索

**RedisSchemaInitializer**：启动时自动创建向量索引（修复 `initialize-schema` 不生效的 bug）

### 检索管线全景

```
用户查询 "如何退款"
    → SearchController.search()
    → SearchService.search(query, topK=3, threshold=0.6)
    → VectorStore.similaritySearch(SearchRequest)
       └── 内部自动调 EmbeddingModel.embed(query)  → 1536 维向量
       └── HNSW 索引 O(log N) 最近邻搜索
    → 返回 List<Document>
    → SearchService.searchAndFormat() → 拼成可读文本
    → API 返回 JSON（含 source / chunkIndex / score / content）
```

## 踩过的坑 / 取舍

### 踩坑 1：Spring AI 2.0.0 不自动创建 FT.INDEX

`initialize-schema=true` 在 Spring Boot 4.x 下不生效 → 检索报 `No such index`。修复：`RedisSchemaInitializer`（CommandLineRunner）启动时检测 + 手动 `FT.CREATE`。

### 踩坑 2：filterExpression 不生效

metadata 存为 JSON 字符串而非独立 hash field → RediSearch FILTER 无法穿透 → 改用内容前缀后置过滤。

### 踩坑 3：threshold 调参无法修复排序

"商品坏了"的正确答案（0.5185）分数低于错误答案（0.5664）→ 调阈值只能同时杀或同时放 → 需要 Week 6 混合检索 + rerank。

### 踩坑 4：metadata 检索时不返回

`similaritySearch` 返回的 Document 只有 `distance` 和 `vector_score`，自定义 metadata（source/chunk_index）丢失 → 修复：灌库时把 metadata 内嵌到内容前缀，检索时用正则提取。

## 面试怎么问

> **"top-K 设多少？为什么？"**

问答场景 3-5，综述场景 8-10。不是越大越好——K 越大 noise 越多，还占 context window。具体值要通过 precision@K 评估来定，不是拍脑袋。

> **"检索结果不相关怎么办？"**

三步排查：1) 降低 threshold 看是否为阈值错杀 2) 检查 query-doc gap（口语 vs 书面语）3) 上 filterExpression 缩小搜索范围。如果还不行 → 混合检索 + rerank（Week 6）。

> **"怎么评估检索质量？"**

构造 10-50 条代表性查询 → 人工标注 ground truth → 跑检索 → 计算 precision@K（top-K 中相关条数 / K）。第一次做不需要 100 条，10 条就能发现明显的 query-doc gap。
