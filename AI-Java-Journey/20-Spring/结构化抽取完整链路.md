# 结构化抽取完整链路

> Week 4 核心产出：MultipartFile → Tika 解析 → 纯文本 → extractV2() → NoteMetadata。知识库"接入层"的完整闭环。

## 它解决什么问题

Day 1-2 的提取端点（`/api/extract/metadata`）需要用户**手动粘贴文本**。真实场景里知识来源是**文件**——你想上传一个 PDF 就得到结构化 JSON。Day 4 把 Day 1-3 的零件串成一条管线的产品形态：

```
┌──────────────┐     ┌──────────────┐     ┌───────────────┐     ┌──────────────┐
│ MultipartFile │ ──▶ │  Tika 解析    │ ──▶ │  extractV2()  │ ──▶ │ NoteMetadata │
│  (PDF/Word/…) │     │  (纯文本)     │     │  (AI 提取)     │     │  (JSON 响应)  │
└──────────────┘     └──────────────┘     └───────────────┘     └──────────────┘
       ↑                    ↑                     ↑                    ↑
   Day 3 前端            Day 3 解析           Day 2 Few-shot         Day 1 合约
```

## 核心要点

### 1. 管线架构

管线的每一段都是**独立可替换**的：

| 管线阶段 | 组件 | 可替换方案 |
|---------|------|-----------|
| 文件接收 | `MultipartFile` | 云存储 URL、Base64 |
| 文档解析 | Apache Tika | PDFBox、POI（需要时单独调用） |
| AI 提取 | `extractV2()` | extract()、自定义 Prompt |
| 结构化输出 | `NoteMetadata` record | 任意自定义 Bean |

**这就像 Unix 管道**——每个阶段的输出是下一阶段的输入，阶段之间不耦合。

### 2. 两条端点路径

```
POST /api/documents/parse
  MultipartFile → Tika → 纯文本
  适用场景：只需要文本，后续处理由调用方自己组合

POST /api/documents/parse-and-extract
  MultipartFile → Tika → 纯文本 → extractV2() → NoteMetadata
  适用场景：一步到位，前端直接拿到结构化 JSON 渲染
```

`/parse` 是原子能力（可被复用），`/parse-and-extract` 是组合能力（便利端点）。

### 3. 五层防跑偏（贯穿整个管线）

| 层 | 机制 | 在哪一阶段 | 作用 |
|----|------|-----------|------|
| 1 | `@JsonProperty(required=true)` | AI 提取 | Schema 强制字段存在 |
| 2 | `@JsonPropertyDescription` | AI 提取 | Schema description 让 LLM 理解语义 |
| 3 | Few-shot 范例 | AI 提取 | 模式补全优于指令跟随 |
| 4 | System 角色约束 | AI 提取 | 规则兜底 |
| 5 | `validateAndFallback()` | AI 提取后 | 后校验 + category 模糊匹配 |

### 4. 工程化防护

```java
// L0：Spring Boot 全局限制（application.properties）
spring.servlet.multipart.max-file-size=10MB

// L1：Controller 前置校验
if (file.isEmpty()) → 400
if (file.getSize() > MAX_FILE_SIZE) → 413

// L2：GlobalExceptionHandler 捕获
MaxUploadSizeExceededException → 413 友好提示

// L3：Service 层异常转换
TikaException → IOException（对外统一异常体系）
```

## Java 里怎么落地

### 完整调用链

```java
// DocumentController.java
@PostMapping(value = "/parse-and-extract", consumes = MULTIPART_FORM_DATA_VALUE)
public ApiResponse<NoteMetadata> parseAndExtract(@RequestParam("file") MultipartFile file) {
    // 前置校验 → 委托 Service → 返回
    if (file.isEmpty()) return ApiResponse.error(400, "文件为空");
    if (file.getSize() > MAX_FILE_SIZE) return ApiResponse.error(413, "文件过大");

    NoteMetadata metadata = parseService.parseAndExtract(
        file.getInputStream(), file.getOriginalFilename());
    return ApiResponse.success(metadata);
}

// DocumentParseService.java
public NoteMetadata parseAndExtract(InputStream is, String filename) throws IOException {
    String text = parse(is, filename);          // Tika：一行搞定所有格式
    return extractService.extractV2(text);      // AI：Few-shot 结构化提取
}
```

## 面试怎么问

- **"从文档上传到结构化输出，整个链路怎么设计的？"** → 四段管线：MultipartFile → Tika（一行 `parseToString` 覆盖 1000+ 格式）→ extractV2（Few-shot + 五层防跑偏）→ NoteMetadata（类型安全 Java Bean）。每段独立可替换。工程化上五层防护：Spring 全局大小限制 → Controller 前置校验 → GlobalExceptionHandler 统一异常 → Service 异常转换 → 后校验兜底。

- **"为什么把 parse 和 parse-and-extract 拆成两个端点？"** → parse 是原子能力（单一职责），可被其他端点复用、组合。parse-and-extract 是便利端点，减少前端调用次数。遵循"原子能力 + 组合能力"的 API 设计原则。

- **"管线如果某一段失败怎么办？"** → 每段有独立异常处理：Tika 解析失败掷 IOException → GlobalExceptionHandler 返回 500。AI 提取失败（API 超时/限流）由 Spring AI Retry 自动重试 3 次 → 仍失败掷异常。后校验阶段 category 跑偏时自动模糊匹配修正（不掷异常，降级兜底）。
