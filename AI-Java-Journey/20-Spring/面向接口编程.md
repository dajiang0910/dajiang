# 面向接口编程

> Service 依赖「接口」而不是「实现」——`NoteService` 注入的是 `NoteRepository`(接口),不是具体实现。这是 DI 的真正威力:**可替换性**。Day 4 用 JPA 替换内存实现,Service 一行不用改。

## 它解决什么问题

如果 Service 直接 `new InMemoryNoteRepository()`,那么:
- 换成 MySQL 实现时,Service 要改代码
- 写单元测时,没法 mock 数据源
- 改一个实现,ripple 到所有调用方

面向接口编程把这些全解了:Service 只知道"这个东西能 `findById`、能 `save`",不关心数据存在内存还是 MySQL。

## 核心要点

```java
// 接口 = 契约:声明「能做什么」
public interface NoteRepository {
    Optional<Note> findById(Long id);
    Note save(Note note);
}

// 实现 = 怎么做,今天内存,明天 MySQL
@Repository
public class InMemoryNoteRepository implements NoteRepository { ... }

// Service 只依赖接口
@Service
public class NoteService {
    private final NoteRepository repo;   // 接口类型,不是实现类型
    public NoteService(NoteRepository repo) { this.repo = repo; }
}
```

**Day 4 实战：从内存到数据库**:
- `NoteRepository` 改为 `extends JpaRepository<Note, Long>`（1 行接口，0 行实现）
- 删除 `InMemoryNoteRepository`（30 行手写实现 → 0 行）
- `NoteService` **一行不用改**——因为它依赖的是 `NoteRepository` 接口

**未来换 MySQL 时**:
- 只改 `application.properties` 的 `datasource.url`
- Service、Controller 都不用改

## 和 DI 的关系

面向接口编程是**思想**,DI 是**落地手段**。没有 DI,你还是得在某处 `new` 具体实现;有了 Spring 的 DI,容器自动把实现注入进来,调用方完全不感知。

## 面试怎么问

- "面向接口编程的好处?" → 解耦、可替换、可 mock 测试。
- "Spring 里怎么实现面向接口?" → 注入时用接口类型声明依赖,Spring 自动找到实现类注入。
