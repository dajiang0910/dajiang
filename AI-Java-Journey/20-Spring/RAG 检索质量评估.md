# RAG 检索质量评估（Week 5 收尾）

> 一句话：量化"检索到底好不好"——不是凭感觉调 topK 和 threshold，而是用 precision@K 数据驱动决策。

## 为什么需要评估

RAG 的核心公理：**RAG 答案质量 ≤ 检索质量**。如果检索没把正确答案捞出来，LLM 再强也白搭。评估告诉我们两件事：
1. 当前检索系统的天花板有多高
2. 哪些查询类型是薄弱点 → Week 6 用混合检索 + rerank 针对性提升

## 评估环境

| 维度 | 取值 |
|------|------|
| 向量库 | Redis Stack（Docker） |
| Embedding 模型 | 百炼 text-embedding-v2（1536 维） |
| LLM | 通义千问 qwen-plus |
| chunk 策略 | TokenTextSplitter chunkSize=800 overlap=100 |
| 评测日期 | 2026-07-02（Week 5 Day 6 收尾） |
| 灌库文档 | refund-policy（退款政策）、after-sales（售后政策）、company-info（公司信息） |

## 测试查询集（手工标注）

每个查询的"正确答案"已事先确定，用于计算 precision@K：

```
precision@K = 前K个检索结果中「相关文档」的数量 / K
```

| # | 查询 | 期望来源 | 难度 | precision@3 |
|---|------|---------|------|-------------|
| Q1 | "如何退款" | refund-policy | 简单 | 3/3（100%） |
| Q2 | "退款政策是什么" | refund-policy | 简单 | 3/3（100%） |
| Q3 | "退款需要什么条件" | refund-policy | 中等 | 2/3（67%） |
| Q4 | "商品坏了怎么办" | after-sales | 困难 | 0/3（0%） ⚠️ |
| Q5 | "公司联系方式" | company-info | 简单 | 3/3（100%） |

**综合 precision@3 = 11/15 ≈ 73.3%**

## 检索失败案例分析

### Q4 "商品坏了怎么办" —— query-doc gap（词汇不匹配）

```
用户查询用词："坏了"（口语化）
文档用词：    "质量问题"（正式用语）
              "产品缺陷"
              "非人为损坏"
```

**gap 类型**：词汇不匹配（vocabulary gap）—— 用户用口语提问，知识库用书面语写文档。

**为什么纯向量检索搞不定？**
- "坏了" 和 "质量问题" 在 Embedding 空间中的余弦相似度不够高
- 百炼 text-embedding-v2 对口语↔书面语的语义桥接能力有限
- 向量检索擅长"语义相似"但不擅长"词汇匹配"

**Week 6 解决方案**：
1. **BM25 关键词检索**：直接匹配"坏"/"损坏"/"故障"等关键词
2. **Query 改写**：LLM 先把"商品坏了怎么办"改写为"产品质量问题如何处理"
3. **混合检索 + rerank**：向量 + BM25 融合 → rerank 精选

### 为什么 Q1-Q3 的 precision 高？

"退款"是通用词汇，用户提问和文档用词高度一致 → 向量检索天然擅长这种场景。

**启示**：当用户提问和文档用词一致时，纯向量检索就够了。当存在词汇 gap 时，需要 Week 6 的混合检索。

## RAG vs 纯 LLM 对比

| 查询 | RAG 答案质量 | 纯 LLM 答案质量 | RAG 提升点 |
|------|------------|---------------|-----------|
| Q1 "如何退款" | ✅ 具体政策：7天无理由、原路返回、3-5工作日 | ⚠️ 通用建议：查看平台政策 | 具体政策细节 + 可追溯来源 |
| Q2 "退款政策" | ✅ 完整政策条款 + 引用片段编号 | ⚠️ 笼统概述，无具体条款 | 准确性 + 可验证 |
| Q3 "退款条件" | ✅ 列出具体条件（时间、状态、凭证） | ⚠️ 泛泛而谈"各平台不同" | 企业专有知识 |
| Q4 "商品坏了" | ❌ 知识库暂无相关信息 | ❌ 编造了一套售后流程 | RAG 至少不说假话 |
| Q5 "公司联系方式" | ✅ 具体电话/邮箱/地址 | ❌ 完全不知道 | 私有知识 |

**核心发现**：RAG 在有知识覆盖的问题上完胜纯 LLM（Q1/Q2/Q3/Q5）；在无知识覆盖的问题上（Q4），RAG 至少不会编造——"安全失败"原则生效。

## 检索天花板验证

```
RAG 答案质量 ≤ 检索质量

Q4 验证了这个公理：
  检索 precision@3 = 0% → Prompt 中无相关上下文
  → RAG 答案只能是"暂无相关信息"
  → 即使 LLM 再强（qwen-plus），也无济于事
```

## 为什么不用框架自带的评估工具

Spring AI 2.0.0 有 `DocumentRetriever` 和 `QuestionAnswerAdvisor` 但没有内置的检索评估模块。手工评估的优势：
1. **透明**：精确知道每个查询的检索结果是什么
2. **可定制**：业务自定义"相关"的定义
3. **面试能讲**：比"我用了一个评估框架"有更多可深挖的细节

Week 6 会引入一个简单的自动化评估脚本（Java 实现），但手工标注的 ground truth 永远是第一步。
