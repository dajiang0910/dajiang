# 多轮对话（消息历史）

> **多轮对话**通过 `ChatMemory` 存储对话历史 + `MessageChatMemoryAdvisor` 自动注入/提取，让 LLM "记住"之前的上下文。同一 `conversationId` 共享记忆，不同 ID 互相隔离。

## 为什么需要消息历史

LLM 本质是**无状态函数**——每次推理从零开始：

```
// Day 1-3 的问题：
POST /api/chat  {"message": "我叫张三"}     → AI: "你好张三！"
POST /api/chat  {"message": "我叫什么名字？"} → AI: "我不知道你的名字" ❌ 失忆了！
```

**根本原因**：每次 `.prompt().user(msg).call()` 都是独立的 HTTP 请求，不携带之前的对话。就像 HTTP 是无状态协议一样。

**解决思路**：把历史消息一起发给 LLM，让它"看到"上下文：

```
[system] 你是智能助手
[user] 我叫张三           ← 第 1 轮（历史）
[assistant] 你好张三！     ← 第 2 轮（历史）
[user] 我叫什么名字？      ← 当前问题（带着历史一起发）
```

## ChatMemory —— 对话记忆的存储

`ChatMemory` 是 Spring AI 的记忆抽象接口：

```java
public interface ChatMemory {
    String CONVERSATION_ID = "chat_memory_conversation_id";  // 上下文参数 key

    void add(String conversationId, List<Message> messages);  // 追加消息
    List<Message> get(String conversationId);                 // 获取历史
    void clear(String conversationId);                        // 清除会话
}
```

**类比**：`ChatMemory` 就像 HTTP Session 存储 —— 按 conversationId 隔离不同用户/会话的数据。

### InMemory 实现

由 `ChatMemoryAutoConfiguration` 自动配置（classpath 中有 `spring-ai-autoconfigure-model-chat-memory` 即生效）：

```java
// 零配置，直接注入即可
@Autowired
private ChatMemory chatMemory;  // InMemory 实现，自动创建的 Bean
```

## MessageChatMemoryAdvisor —— Advisor 模式

**Advisor 模式** 是 Spring AI 的设计精髓 —— 类似 Servlet Filter / Spring Interceptor，在 LLM 调用前后织入横切逻辑。

```
请求 → [Advisor.before: 注入历史] → LLM 推理 → [Advisor.after: 保存本轮] → 响应
```

### 工作原理

```java
MessageChatMemoryAdvisor advisor = MessageChatMemoryAdvisor.builder(chatMemory).build();

// before(): 从 ChatMemory 取出 conversationId 的历史消息 → 注入 Prompt
// after():  把本轮 user + assistant 消息存入 ChatMemory
```

| 时机 | 动作 | 效果 |
|---|---|---|
| **请求前** (before) | `chatMemory.get(conversationId)` → 将历史消息注入 Prompt | LLM "看到"之前的对话 |
| **响应后** (after) | `chatMemory.add(conversationId, [userMsg, assistantMsg])` | 本轮对话保存到记忆 |

## conversationId —— 对话的"身份证"

- **同一 conversationId** → 共享历史记忆（AI 能记住）
- **不同 conversationId** → 互相隔离（全新对话）
- 类比：HTTP Session ID / 数据库的 partition key

```java
// 会话 A：记住名字
POST /api/chat/multi-turn  {"conversationId": "conv-001", "message": "我叫张三"}
POST /api/chat/multi-turn  {"conversationId": "conv-001", "message": "我叫什么名字？"}
// → "你叫张三" ✅

// 会话 B：全新对话
POST /api/chat/multi-turn  {"conversationId": "conv-002", "message": "我叫什么名字？"}
// → "我不知道你的名字"（隔离生效）
```

## 代码实战

### ChatService —— 注入 ChatMemory + Advisor 配置

```java
@Service
public class ChatService {
    private final ChatClient chatClient;
    private final ChatMemory chatMemory;  // Day 4 新增：自动配置的 InMemory Bean

    public ChatService(ChatClient.Builder builder, ChatMemory chatMemory) {
        this.chatClient = builder.build();
        this.chatMemory = chatMemory;
    }

    public String chatMultiTurn(String conversationId, String userMessage) {
        return chatClient.prompt()
                .advisors(a -> a
                    .param(ChatMemory.CONVERSATION_ID, conversationId)  // 设置会话 ID
                    .advisors(MessageChatMemoryAdvisor.builder(chatMemory).build()))  // 注册 Advisor
                .user(userMessage)
                .call()
                .content();
    }
}
```

### `AdvisorSpec` 链式配置解读

```java
.advisors(a -> a                                    // AdvisorSpec: 配置入口
    .param(ChatMemory.CONVERSATION_ID, "conv-001")  // 设上下文参数 → Advisor 用它查历史
    .advisors(MessageChatMemoryAdvisor.builder(chatMemory).build())  // 注册 Advisor Bean
)
```

- `param(key, value)` → 放入 advisor 上下文 → `MessageChatMemoryAdvisor` 的 `getConversationId()` 从中提取
- `advisors(Advisor...)`  → 注册 Advisor 到执行链

### ChatController —— 多轮端点

```java
@PostMapping("/chat/multi-turn")
@Operation(summary = "多轮对话", description = "携带对话历史的多轮对话")
public ApiResponse<String> chatMultiTurn(@Valid @RequestBody ChatMultiTurnRequest request) {
    return ApiResponse.success(
        chatService.chatMultiTurn(request.conversationId(), request.message()));
}
```

### 请求体 DTO

```java
public record ChatMultiTurnRequest(
    @NotBlank String conversationId,  // Day 4 新增字段：会话标识
    @NotBlank String message          // 用户消息（和单轮一样）
) {}
```

## Advisor 模式 vs Filter/Interceptor

| | Servlet Filter | Spring Interceptor | MessageChatMemoryAdvisor |
|---|---|---|---|
| 作用域 | HTTP 请求 | Spring MVC | LLM 调用 |
| before | 请求到达前 | Handler 执行前 | Prompt 发送前 |
| after | 响应返回前 | Handler 执行后 | LLM 响应后 |
| 横切逻辑 | 日志/鉴权/CORS | 日志/权限 | 历史注入/保存 |

## 面试怎么问

- "LLM 多轮对话怎么实现？" → LLM 无状态，每次请求把历史消息一起发。Spring AI 用 ChatMemory 存历史 + MessageChatMemoryAdvisor 自动注入/提取。
- "Advisor 模式是什么？" → Spring AI 的拦截器机制，在 LLM 调用前后织入横切逻辑。类比 Servlet Filter。
- "conversationId 的作用？" → 区分不同对话会话，同一 ID 共享历史，不同 ID 互相隔离。类似 HTTP Session ID。
- "ChatMemory 的 Bean 从哪来？" → ChatMemoryAutoConfiguration 自动配置，classpath 有 spring-ai-autoconfigure-model-chat-memory 即创建 InMemory Bean。
