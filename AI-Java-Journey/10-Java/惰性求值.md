# 惰性求值

> Stream 的中间操作(filter/map/...)只在"画图纸",遇到终端操作(collect/count/forEach)才"启动流水线"真正执行。

## 它解决什么问题
理解"为什么链式写一堆方法时,网络/文件/DB 还没被访问"——因为还没人按"启动键"。

## 它怎么工作
- **中间操作**(filter/map/distinct/...):画图纸,不执行
- **终端操作**(collect/count/forEach):启动!真正跑

## 一行验证(2 步看清楚)
```java
docs.stream()
    .peek(d -> System.out.println("经过 filter 前:" + d.title()))
    .filter(d -> d.category().equals("财务"))
    .peek(d -> System.out.println("经过 filter 后:" + d.title()))
    .count();   // 👈 加上才看到上面所有打印
```
- 删掉 `.count()`:程序什么也不输出
- 加上 `.count()`:所有打印出现

## 实际意义
- 读 Spring AI / RAG 代码时常见:`chatClient.prompt().advisors(...).tools(...).call()` —— 中间所有配置都不发请求,`.call()` 才是真正发送。
- 能把多个中间操作"合并"再执行,JVM 还能做些优化(短路、合并遍历)。

## 面试怎么问(高频)
- "Stream 是不是立刻执行?" → 不是,中间操作惰性,遇终端才执行。
- "什么是中间操作?什么是终端操作?" → 上面那张表对照答。
